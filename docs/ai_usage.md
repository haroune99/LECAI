# AI Usage Note

## What I Built vs What the Coding Agent Wrote

This note describes which parts of the system reflect my architectural decisions and which were generated by the coding agent — and how I verified the output.

## My Decisions (Architecture and Design)

The following are entirely mine — chosen and justified by me:

1. **LangGraph over raw Python:** I evaluated both and chose LangGraph because the conditional edge routing and state checkpointing handle retry logic that would otherwise require significant custom code. The rejection of LangChain was deliberate — its abstraction layer would obscure the decision logic in a 30-minute walkthrough.

2. **MiniMax-M2.7 as primary model:** I researched MiniMax's API documentation, confirmed tool calling support, and designed the integration with `reasoning_split=True` to handle the `<think>...</think>` tag issue. The cost model ($0.30/M input, $1.20/M output) and the corrected per-query cost of $0.00177 are my calculations.

3. **5-tool design:** Each tool was designed to solve a coherent LEC business problem. The Trade Calculator's landed cost formula reflects LEC Beverages' actual import workflow. The Partnership Profiler's cross-reference to sanctions screening reflects LEC Global Capital's compliance requirements.

4. **Eval harness structure:** The 10 queries, grading rubric (0/1/2), and LLM-as-judge prompt were designed by me. The prompt ablation structure (v1 structured vs v2 loose) is my evaluation methodology.

5. **Streamlit password protection:** The password gate implementation is mine — it was a deliberate production-awareness signal (showing I thought about sharing a URL publicly).

## Coding Agent Generation

The following were generated by the coding agent following my specifications:

- **All Python source files:** The coding agent wrote the implementation of the components I designed — the LangGraph nodes, the tool functions, the retrieval pipeline, the eval harness, the Streamlit app. I provided the specifications; the agent wrote the code.

- **Repository scaffolding:** Folder structure, `pyproject.toml`, `.env.example`, `.gitignore` — generated by the coding agent from a specification document.

- **Test files:** `test_trade_calculator.py`, `test_trade_regulations.py`, `test_budget.py`, `test_agent_nodes.py`, `test_retrieval.py` — generated by the coding agent to cover the components it wrote.

## Verification

I verified the coding agent's output through:

1. **Unit tests:** Each tool has at least one unit test. The budget tracker has a test that confirms it fires correctly at the cap. The planner node has a test that confirms it parses the structured format correctly.

2. **Manual code review:** I reviewed key files — `planner.py`, `executor.py`, `graph.py` — to confirm the routing logic matched the architecture specification.

3. **The eval harness:** Running the full eval harness (when API keys are available) produces real numbers. I designed the eval structure to catch hallucination and tool failure — if the agent consistently fails a query type, the scores reveal it.

## Where I'd Push Harder with More Time

If I had another week, the two things I'd most want to verify myself rather than delegate to the coding agent:

1. **The planner prompt v1 parsing.** The `parse_plan_steps` function is hand-rolled regex parsing of the planner's structured output. It's brittle — if the planner changes the format slightly, it breaks silently. With more time I'd write a more robust parser or use JSON-mode prompting.

2. **The retrieval pipeline's English-split logic for the Tsingtao PDF.** The regex `[a-zA-Z]` vs total chars ratio is a heuristic. I'd want to verify coverage manually on a sample of pages and potentially add OCR as a fallback.
